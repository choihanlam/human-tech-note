AI 시대의 보안 자산은 무엇인가

보안 자산의 경계가 달라졌다

오늘날 조직의 보안 자산은 더 이상 서버, 데이터베이스, 네트워크 장비, 개인정보 처리 시스템에만 머물지 않는다. AI Agent가 업무를 수행하기 위해 접근하고, 기억하고, 호출하고, 변경할 수 있는 모든 데이터, 권한, 도구와 행위 경로가 새로운 보안 자산이 되고 있다.

AI 보안 거버넌스의 출발점
우리 조직은 AI가 접근할 수 있는 것을 자산으로 관리하고 있는가, 아니면 여전히 장비와 시스템만 자산으로 보고 있는가?

이 질문은 단순히 자산 목록을 더 많이 만들자는 의미가 아니다. 핵심은 조직이 실제 공격 표면을 어디까지 인식하고 있는가에 있다. 과거에는 서버, DB, 네트워크 장비, 업무시스템, 개인정보 처리 시스템처럼 비교적 명확한 단위로 자산을 식별할 수 있었다.

그러나 AI Agent, SaaS, API, 오픈소스 저장소, 클라우드 리소스와 자동화 워크플로가 결합되는 환경에서는 자산의 경계가 더 이상 고정되어 있지 않다.

자산 경계 비교적 고정된 시스템과 저장 위치 API와 SaaS를 따라 동적으로 연결되는 실행 경로
통제 질문 어디에 저장되어 있고 누가 접근하는가 어떤 Agent가 어떤 맥락과 권한으로 무엇을 실행하는가

Agent는 새로운 행위 주체다

AI Agent는 사람처럼 로그인하지 않을 수 있지만, 사람을 대신해 업무를 수행한다. 파일을 읽고, 데이터를 요약하고, API를 호출하며, 외부 도구를 실행하고, GitHub 저장소와 SaaS 계정을 참조한다. 경우에 따라 코드를 수정하거나 배포 흐름에 영향을 줄 수도 있다.

이때 보안상 중요한 것은 “AI 모델이 존재하는가”가 아니다. AI가 어떤 권한으로 무엇에 접근하고 어떤 행위를 할 수 있는가이다.

핵심 관점

AI 시대의 자산 식별은 모델 목록을 만드는 일이 아니라, Agent가 실제 업무를 수행하는 실행 경로 전체를 식별하는 일이다.

새롭게 관리해야 할 열 가지 보안 자산

AI 환경에서 자산의 범위가 확장된다는 것은 기존 자산관리 체계를 버려야 한다는 뜻이 아니다. 기존 통제를 Agent의 신원, 권한, 기억, 연결과 행위까지 확장해야 한다는 의미다.

01

AI Agent의 실행 권한과 비인간 ID

기존 접근통제는 주로 사람, 부서와 시스템 계정을 중심으로 설계됐다. 그러나 AI Agent와 자동화 봇은 조직 내부에서 새로운 행위 주체로 작동한다. 이들은 사람이 아니지만 업무를 실행하고 데이터를 조회하며 외부 시스템을 호출한다. 따라서 Agent ID, Bot 계정, Service Account와 자동화 계정은 단순한 운영 편의 기능이 아니라 독립적인 보안 자산으로 관리해야 한다.

이 Agent의 소유자, 목적, 유효기간과 권한 회수 조건을 설명할 수 있는가?
02

API Token, OAuth Token, Service Account Key

AI Agent는 외부 도구나 내부 시스템과 연동하기 위해 토큰과 키를 사용한다. 자격증명이 과도한 권한을 갖거나 장기간 갱신되지 않은 채 방치돼 있다면, 공격자는 AI 자체를 해킹하지 않더라도 Agent가 사용하는 권한을 탈취해 조직 내부 시스템에 접근할 수 있다. 토큰은 단순 인증 수단이 아니라 조직의 권한 경계를 결정하는 자산이다.

자격증명의 Scope, 저장 위치, 만료일, 회전주기와 사용 이력이 관리되고 있는가?
03

MCP 서버와 외부 Tool

AI Agent는 검색, 파일 처리, 코드 실행, 업무시스템 조회, 클라우드 제어와 협업도구 연동 등 다양한 외부 Tool을 호출한다. MCP와 같은 연결 구조가 확산될수록 Agent가 어떤 Tool을 사용할 수 있는지, 그 Tool이 어떤 권한을 갖는지, 신뢰 가능한 출처에서 운영되는지가 중요한 보안 쟁점이 된다. Tool은 생산성 도구이면서 동시에 공격 경로가 될 수 있다.

허용된 Tool 목록, 운영 주체, 입력 검증, 민감 행위 승인과 호출 제한이 정의돼 있는가?
04

데이터셋, 벡터DB와 임베딩

AI 시스템은 문서와 데이터를 단순히 저장하지 않고 검색·요약·추론 가능한 형태로 재구성한다. 벡터DB와 임베딩에는 원문 데이터의 의미 구조가 반영될 수 있으며 업무 지식, 고객 정보, 내부 정책, 기술 문서와 코드 조각이 포함될 수 있다. 따라서 벡터DB는 단순한 보조 저장소가 아니라 조직 지식이 압축된 민감 자산으로 봐야 한다.

원문과 임베딩의 권한, 보존기간, 삭제와 재색인 기준이 함께 관리되는가?
05

System Prompt, Agent Memory와 업무 Context

AI Agent가 어떤 역할로 설정돼 있는지, 어떤 정책을 따르도록 설계됐는지, 이전 업무 맥락을 어디까지 기억하는지는 Agent의 행위에 직접 영향을 준다. System Prompt와 Memory에는 내부 업무 방식, 의사결정 기준, 접근 가능한 도구, 사용자 선호와 조직의 암묵지가 포함될 수 있다. 프롬프트와 메모리는 단순 텍스트가 아니라 Agent의 행위를 형성하는 통제 자산이다.

Prompt와 Memory의 변경 권한, 민감정보 포함 여부, 무결성과 초기화 절차가 관리되는가?
06

SaaS 계정과 자동화 Workflow

Slack, Microsoft 365, Google Workspace, Jira, Confluence, GitHub와 Notion 같은 SaaS는 조직 업무의 기본 인프라가 됐다. 여기에 AI Agent가 연결되면 일정 생성, 문서 작성, 파일 공유, 티켓 처리, 코드 리뷰와 권한 변경 같은 행위가 자동화된다. SaaS 계정과 Workflow는 업무 효율화 수단이면서 오남용 시 대규모 정보유출과 권한 확산을 유발할 수 있는 자산이다.

Workflow가 실행할 수 있는 쓰기·공유·삭제·권한변경 행위와 중단 방법이 정의돼 있는가?
07

GitHub 저장소, CI/CD와 Cloud Resource

AI Agent가 개발 환경에 연결되면 보안 자산의 범위는 소스코드 저장소와 배포 파이프라인까지 확장된다. GitHub 저장소, Actions Secret, CI/CD 토큰, 컨테이너 이미지와 클라우드 배포 권한은 모두 Agent의 행위와 연결될 수 있다. 악성 저장소, 오염된 패키지와 잘못된 자동 배포 설정은 Agent를 통해 더 빠르게 조직 내부로 확산될 수 있다.

Agent가 코드 수정, Merge, Secret 조회와 배포를 어디까지 수행할 수 있는가?
08

Tool Calling과 Agent 행위 로그

AI Agent가 어떤 입력을 받았고 어떤 도구를 호출했으며, 어떤 데이터에 접근해 어떤 결과를 생성했는지는 사고 분석과 책임 규명에 필수적이다. 로그는 단순 저장 대상이 아니라 조직이 Agent의 행위를 설명하기 위한 근거다. AI 환경에서는 “누가 실행했는가”뿐 아니라 “어떤 Agent가, 어떤 맥락에서, 어떤 도구를 통해, 어떤 권한으로 실행했는가”를 추적할 수 있어야 한다.

사용자 요청부터 Tool 입력·출력, 승인, 변경 결과까지 하나의 사건으로 연결해 재구성할 수 있는가?
09

모델·프롬프트·데이터·Tool 버전

AI 시스템의 결과는 모델 버전, 프롬프트 버전, 데이터 버전과 Tool 설정에 따라 달라질 수 있다. 같은 질문이라도 어떤 모델과 데이터, 어떤 프롬프트 체인을 사용했는지에 따라 다른 결과가 나온다. 따라서 AI 시스템의 변경 이력은 단순한 개발 관리 항목이 아니라 보안성과 책임성을 검증하기 위한 자산이다.

사고 당시 사용된 모델, Prompt, 지식 원천과 Tool 설정을 동일하게 식별할 수 있는가?
10

자산 소유자와 위험 승인 책임

AI Agent가 접근하는 데이터와 도구가 많아질수록 누가 권한을 승인했고, 누가 위험을 수용했으며, 문제가 발생하면 누가 설명할 수 있는지가 중요해진다. 자산관리의 핵심은 목록화가 아니라 소유권과 책임성을 명확히 하는 것이다. 소유자가 없는 자산, 승인자가 불명확한 권한과 위험 수용 기록이 없는 자동화는 모두 조직의 보이지 않는 공격 표면이 된다.

Business Owner, Technical Owner, Data Owner와 Risk Approver가 구분돼 있는가?

자산대장에서 통제모델로

결국 AI 시대의 보안은 완전히 새로운 보안이라기보다 기존 보안 통제의 분석 단위를 재정의하는 문제에 가깝다. 자산 식별, 접근통제, 로그관리, 외부입력 검증과 사고대응이라는 기본 통제는 여전히 유효하다. 다만 그 적용 대상이 서버와 DB에서 Agent의 권한, 행위, 기억과 연결 구조로 확장됐다.

조직이 바로 확인할 질문

AI 보안 거버넌스의 핵심은 AI를 무조건 차단하는 것이 아니다. AI가 조직 내부에서 어떤 행위 주체로 작동하고 있는지, 어떤 자산에 접근하는지, 어떤 권한으로 외부 도구를 호출하는지, 그리고 조직이 그 과정을 설명할 수 있는지를 관리하는 것이다.

  • 승인된 AI Agent와 비승인 Agent를 구분할 수 있는가?
  • Agent가 사용하는 모든 Token과 Service Account를 식별했는가?
  • MCP 서버와 외부 Tool의 운영 주체와 신뢰수준이 검증됐는가?
  • 벡터DB와 임베딩에도 원문 데이터의 등급과 권한이 연결되는가?
  • Prompt와 Memory에 개인정보·기밀정보가 남는 범위를 알고 있는가?
  • Agent가 SaaS에서 수행할 수 있는 쓰기·공유·삭제 행위를 제한했는가?
  • 코드 수정과 배포에는 사람의 승인 또는 별도 통제가 존재하는가?
  • Tool 호출과 실제 시스템 변경 결과를 하나의 로그로 연결할 수 있는가?
  • 모델·Prompt·Data·Tool 변경이 승인과 버전관리 대상인가?
  • 문제가 발생하면 Agent를 중단하고 권한을 즉시 회수할 수 있는가?

AI 활용이 통제 가능한 조직 역량이 되려면

AI가 등장했다고 해서 보안의 본질이 사라진 것은 아니다. 오히려 보안의 기본 개념인 자산, 권한, 로그, 신뢰와 책임이 더 정교하게 요구되고 있다. 보호해야 할 자산의 범위는 넓어졌고 권한의 단위는 더 세밀해졌으며, 사고 이후 조직이 져야 할 설명 책임은 더 무거워졌다.

최종 질문
AI는 우리 조직의 어떤 데이터와 도구에 접근하고 있으며, 그 접근은 어떤 목적과 권한에 의해 승인되었고, 문제가 발생했을 때 조직은 그 과정을 설명할 수 있는가?

이 질문에 답할 수 있을 때 AI 활용은 통제 가능한 조직 역량이 된다. 반대로 답할 수 없다면, AI는 생산성 도구가 아니라 조직 내부에 숨어 있는 새로운 공격 표면으로 남게 된다.

Human Tech Note의 관점

AI 시대의 보안 자산 관리는 자산 목록의 확장이 아니라, AI의 신원·권한·기억·연결·행위와 책임을 하나의 통제 흐름으로 관리하는 것이다.
